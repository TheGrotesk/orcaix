AI as Code

Define AI agent workflows in YAML. Chain LLMs, shell commands, file operations, and HTTP calls into repeatable pipelines — with human approval gates, branching logic, and built-in resume from failure.

🧩
6 Stage Types
LLM, shell, file, HTTP, input, and loop — compose anything.
🤖
3 Providers
Anthropic, OpenAI, and Gemini — mix models in a single workflow.
🔀
Conditional Routing
Branch on any JS expression evaluated against stage output.
💾
Checkpoints
Auto-resume from the last completed stage after any failure.

Installation

bash
npm install -g @thegrotesk/aiac

Set your provider API keys as environment variables, or put them in a .env file in the directory you run aiac from:

.env
ANTHROPIC_API_KEY=sk-ant-...
OPENAI_API_KEY=sk-...
GEMINI_API_KEY=AIza...

Quick Start

Create a workflow file, or scaffold one with aiac init:

bash
aiac init my-workflow        # scaffold a starter YAML
aiac validate my-workflow.yaml   # check for schema errors
aiac run my-workflow.yaml --prompt "your input here"

Here's the simplest possible workflow — a single LLM call:

hello.yaml
name: Hello World
stages:
  - id: greet
    type: llm
    provider: anthropic
    model: claude-haiku-4-5-20251001
    prompt: "Say hello to: {{input}}"
    max_tokens: 100
bash
aiac run hello.yaml --prompt "Alice"
# → Hello, Alice! It's wonderful to meet you.

Workflow Structure

A workflow is a YAML file with the following top-level fields:

FieldTypeDescription
namerequiredHuman-readable workflow name, shown in the terminal.
descriptionoptionalMulti-line description printed at the start of a run.
versionoptionalFreeform version string (e.g. "1.0").
importoptionalArray of paths to template YAML files to import. Resolved relative to the workflow file.
templatesoptionalInline key→string map of reusable prompt snippets. Overrides imported templates on conflict.
stagesrequiredOrdered list of stage objects. Must have at least one.
workflow.yaml — full structure
name: My Workflow
description: |
  What this workflow does.
version: "1.0"

import:
  - ./templates/personas.yaml
  - ./templates/formats.yaml

templates:
  my_persona: You are a helpful expert. Be concise.

stages:
  - id: step1
    type: llm
    # ...
  - id: step2
    type: shell
    # ...

Execution model

Stages run linearly by default — each stage's output becomes the next stage's {{input}}. You can override this with next conditions to branch or jump to any stage by ID.

stage 1
→
stage 2
→
stage 3
→
output

All outputs are stored in ctx.stages and remain accessible via interpolation for the entire run — regardless of branching.

Interpolation

Every string field in a workflow supports {{variable}} placeholders. They are resolved at execution time using the current context.

{{input}}
The output of the previous stage (or the initial --prompt for the first stage).
{{stages.id.output}}
Output of any previously completed stage by its id. Available throughout the entire run, even after branching.
{{env.VAR_NAME}}
Environment variable. Reads from the process environment (including any .env file loaded with --env-file).
{{vars.KEY}}
Variables passed via --var KEY=value on the CLI, or set internally by the loop stage (loop_iteration, loop_feedback).
{{templates.name}}
Inline or imported template snippet by key. Expands to the full multi-line string.
{{timestamp}}
ISO timestamp of when the run started, with : and . replaced by -. Useful for unique filenames.

Loop-specific variables

These are automatically set when inside a loop stage:

{{vars.loop_iteration}}
Current iteration number (1-based). Available in all sub-stage prompts and the approve_message.
{{vars.loop_feedback}}
Feedback text typed by the user during the approval prompt. Empty on the first iteration. Cleared automatically when the loop exits.

Templates

Templates are reusable prompt snippets referenced with {{templates.name}}. They can be defined inline in the workflow or imported from separate files.

Inline templates

yaml
templates:
  senior_engineer: |
    You are a senior software engineer.
    Write minimal, targeted, production-ready code.
  output_format: |
    Output ONLY file content using === FILE: path === blocks.
    No prose, no explanations.

stages:
  - id: implement
    type: llm
    provider: anthropic
    model: claude-sonnet-4-6
    system: "{{templates.senior_engineer}}\n{{templates.output_format}}"
    prompt: "Implement: {{input}}"

Imported template files

Create a standalone YAML file with only a templates: block:

templates/personas.yaml
templates:
  senior_engineer: You are a senior software engineer...
  code_reviewer: You are an expert code reviewer...
  sre: You are a Site Reliability Engineer...

Then import it in your workflow:

yaml
import:
  - ./templates/personas.yaml
  - ./templates/formats.yaml
ℹ️
Merge precedence Later imports override earlier ones on key conflict. Inline templates: in the workflow always wins.

Conditional Routing

Any stage can have a next array to route execution to a specific stage based on the output. Conditions are JavaScript expressions evaluated with output in scope.

yaml
- id: classify
  type: llm
  provider: anthropic
  model: claude-haiku-4-5-20251001
  system: "Reply with exactly one word: critical, warning, or ok"
  prompt: "Classify this: {{input}}"
  max_tokens: 10
  next:
    - condition: "output.toLowerCase().includes('critical')"
      stage: escalate
    - condition: "output.toLowerCase().includes('warning')"
      stage: investigate
    - stage: close_ticket    # no condition = default fallback

Rules:

  • Conditions are evaluated in order — first match wins.
  • A next entry without a condition is the default fallback — it matches anything.
  • If no condition matches and there's no default, execution continues linearly to the next stage.
  • The output variable in the expression is the current stage's output string.
⚠️
Skipped stages are not lost. Jumping from stage 2 to stage 5 doesn't erase stages 3–4 — they simply never run. All completed stage outputs remain accessible via {{stages.id.output}}.

Checkpoints

aiac automatically saves a checkpoint after every completed stage. If the workflow fails or is interrupted mid-run, you can resume from where it stopped — saving expensive LLM calls.

Checkpoints are stored in .aiac-checkpoints/ in the current working directory, keyed by a hash of the workflow path + prompt. Each checkpoint records completed stage outputs, current vars, and the next stage index.

Resume behavior

The next time you run the same workflow with the same prompt, aiac detects the checkpoint:

terminal
Checkpoint found — started 2024-01-15T10:32:11, last updated 2024-01-15T10:38:44
  Completed stages: analyze, approve_plan, create_plan

Resume from checkpoint? (yes / no — "no" starts fresh) _

CLI flags

FlagBehavior
--resumeSilently resume from checkpoint without prompting.
--freshIgnore any existing checkpoint and start from the beginning.
(none)Interactively ask if a checkpoint exists.

On successful completion, the checkpoint is automatically deleted.

Stage: llm

Calls an LLM provider and returns the response text as the stage output.

FieldStatusTypeDescription

modelrequiredstringModel ID, e.g. claude-sonnet-4-6, gpt-4o, gemini-2.0-flash.
promptrequiredstringThe user message. Supports interpolation.

Injecting codebase context with files

The files array accepts glob patterns. Matched files are prepended to the prompt as XML blocks, giving the model full codebase context:

yaml
- id: audit
  type: llm
  provider: anthropic
  model: claude-opus-4-7
  files:
    - src/**/*.ts
    - src/**/*.tsx
    - package.json
  prompt: |
    Audit the codebase above for security vulnerabilities.
    Feature request: {{input}}

Files are injected in this format:

injected context format
<file path="src/auth.ts">
... file content ...
</file>
<file path="package.json">
... file content ...
</file>
⚠️
Size limits A warning is logged when total file content exceeds 100 KB. An error is thrown above 500 KB to prevent silent context truncation.

Stage: shell

Runs a shell command. stdout becomes the stage output. stderr is passed through but doesn't fail the stage.

FieldStatusTypeDescription
commandrequiredstringShell command. Supports interpolation. Multi-line YAML block scalars work.
workdiroptionalstringWorking directory for the command. Defaults to process CWD.
interactiveoptionalbooleanIf true, spawns the process with stdio: inherit — the subprocess takes over the terminal. Workflow resumes when the process exits.
nextoptionalNextCondition[]Conditional routing on stdout output.

All environment variables (including those from .env) are available in shell commands. Stage variables can be injected via interpolation:

yaml
- id: typecheck
  type: shell
  command: npx tsc --noEmit 2>&1 || true

- id: deploy
  type: shell
  command: |
    echo "Deploying to {{env.DEPLOY_ENV}}..."
    git push origin main
    curl -X POST {{env.DEPLOY_WEBHOOK}}

Interactive mode

Set interactive: true to hand off the terminal to the subprocess. The workflow pauses until the process exits:

yaml
- id: implement
  type: shell
  interactive: true
  # Claude Code opens interactively. Type /exit when done to continue the workflow.
  command: claude --model sonnet "$(cat /tmp/plan.txt)"
💡
Use interactive: true with Claude Code to get a fully interactive implementation session mid-workflow. The workflow resumes automatically after you exit.

Passing LLM output to shell commands

Avoid shell-escaping issues (backticks, quotes) by saving LLM output to a temp file first:

yaml
- id: save_output
  type: file
  action: write
  path: /tmp/llm_output.txt
  content: "{{stages.implement.output}}"

- id: apply
  type: shell
  command: |
    node -e "
      const fs = require('fs'), path = require('path');
      const text = fs.readFileSync('/tmp/llm_output.txt', 'utf-8');
      const parts = text.split(/^=== FILE: (.+?) ===$/m);
      for (let i = 1; i < parts.length; i += 2) {
        const filePath = parts[i].trim();
        const content = (parts[i+1] || '').replace(/^\n/, '');
        fs.mkdirSync(path.dirname(filePath), { recursive: true });
        fs.writeFileSync(filePath, content, 'utf-8');
      }
    "

Stage: loop

Runs a sub-pipeline repeatedly until a condition is met or the user approves. Sub-stages share the parent context — outputs are written to ctx.stages and persist across iterations.

FieldStatusTypeDescription
stagesrequiredStage[]Sub-stages to run each iteration. Can include any stage type, including nested loops.
untilrequiredJS expressionExit condition. Evaluated after each iteration with output (last sub-stage output) and stages in scope. Use "false" for approval-only loops.
max_iterationsoptionalintegerMaximum iterations before stopping. Default: 10.
approveoptionalbooleanIf true, pause after each iteration for human review.
approve_messageoptionalstringWhat to display during the approval prompt. Supports interpolation. Defaults to the last sub-stage output.
nextoptionalNextCondition[]Conditional routing after the loop completes.

Approval responses

When approve: true, the user sees the approve_message and a prompt with three options:

InputEffect
done, ship, yes, y, ok, approve, lgtmAccept — exit the loop, continue the workflow.
stop, quit, abort, exit, cancel, noAbort — exit the loop immediately, no further iterations.
any other textFeedback — stored in {{vars.loop_feedback}} and used as input for the next iteration's sub-stages.

Example: Fix loop until tests pass

yaml
- id: fix_loop
  type: loop
  max_iterations: 6
  until: "!stages.run_tests.output.toLowerCase().includes('fail')"
  stages:
    - id: analyze
      type: llm
      provider: anthropic
      model: claude-sonnet-4-6
      files:
        - src/**/*.ts
        - tests/**/*.ts
      prompt: |
        Failing tests:
        {{stages.run_tests.output}}

        Fix the source files. Output as === FILE: path === blocks.
      max_tokens: 4096

    - id: apply
      type: shell
      command: node scripts/apply-blocks.js

    - id: run_tests
      type: shell
      command: npm test 2>&1 || true

Example: Iterative development with approval

yaml
- id: dev_loop
  type: loop
  max_iterations: 5
  approve: true
  approve_message: |
    ── Iteration {{vars.loop_iteration}} ────────────────────────────

    TYPECHECK:  {{stages.typecheck.output}}
    TESTS:      {{stages.run_tests.output}}
  until: "false"   # driven entirely by human approval
  stages:
    - id: implement
      type: llm
      provider: anthropic
      model: claude-sonnet-4-6
      prompt: |
        Plan: {{stages.create_plan.output}}
        Feedback: {{vars.loop_feedback}}
        Typecheck errors: {{stages.typecheck.output}}
      max_tokens: 8000

    - id: typecheck
      type: shell
      command: npx tsc --noEmit 2>&1 || true

    - id: run_tests
      type: shell
      command: npm test 2>&1 || true

Pattern: Plan → Approve → Build

The most common agentic pattern. An expensive model produces a plan, a human reviews and optionally refines it, then a cheaper or more specialized model executes it.

Opus: analyze
→
human: approve
→
Opus: plan
→
human: approve
→
Sonnet: implement
yaml — two-phase approval
stages:
  - id: analyze
    type: llm
    provider: anthropic
    model: claude-opus-4-7
    files: [src/**/*.ts, package.json]
    prompt: "Feature: {{input}}\nAnalyze the codebase and define acceptance criteria."

  - id: approve_analysis
    type: input
    message: "{{input}}\n\nType 'yes' to proceed or give feedback:"
    next:
      - condition: "output.toLowerCase().trim() === 'yes'"
        stage: create_plan
      - stage: refine_analysis

  - id: refine_analysis
    type: llm
    provider: anthropic
    model: claude-opus-4-7
    prompt: |
      Original: {{stages.analyze.output}}
      Feedback: {{input}}
      Revise accordingly.

  - id: create_plan
    # ... implementation follows

Pattern: Autonomous Fix Loop

Run tests, analyze failures, apply fixes, repeat until green — no human in the loop. Set a sane max_iterations to cap spend.

run tests
→
check: passing?
→ no →
LLM: fix
→
apply
→ loop
yaml
stages:
  - id: initial_run
    type: shell
    command: npm test 2>&1 || true

  - id: gate
    type: llm
    provider: anthropic
    model: claude-haiku-4-5-20251001
    system: "Reply with one word: passing or failing"
    prompt: "{{input}}"
    max_tokens: 5
    next:
      - condition: "output.includes('passing')"
        stage: done
      - stage: fix_loop

  - id: fix_loop
    type: loop
    max_iterations: 6
    until: "!stages.run_tests.output.toLowerCase().includes('fail')"
    stages:
      - id: fix
        type: llm
        provider: anthropic
        model: claude-sonnet-4-6
        files: [src/**/*.ts, tests/**/*.ts]
        prompt: "Fix these failures:\n{{stages.run_tests.output}}"
        max_tokens: 4096
      - id: apply
        type: shell
        command: node scripts/apply-blocks.js
      - id: run_tests
        type: shell
        command: npm test 2>&1 || true

  - id: done
    type: llm
    provider: anthropic
    model: claude-haiku-4-5-20251001
    prompt: "Summarize test results: {{stages.run_tests.output}}"

Pattern: Multi-Provider Pipeline

Use the best model for each task. Expensive models for reasoning, cheap models for classification and formatting, specialized models for code.

yaml — analyze (Opus) → suggest (GPT-4o) → classify (Haiku)
stages:
  - id: analyze
    type: llm
    provider: anthropic
    model: claude-opus-4-7       # deep reasoning
    prompt: "Analyze: {{input}}"

  - id: suggest
    type: llm
    provider: openai
    model: gpt-4o               # strong code generation
    prompt: "Suggest fixes for:\n{{input}}"

  - id: classify
    type: llm
    provider: anthropic
    model: claude-haiku-4-5-20251001   # fast, cheap triage
    system: "Reply with one word: critical, warning, or ok"
    prompt: "{{stages.analyze.output}}"
    max_tokens: 10
    next:
      - condition: "output.includes('critical')"
        stage: alert

Pattern: Human Approval Gate

Insert an input stage anywhere in the pipeline to get human sign-off before a destructive or expensive operation.

yaml
- id: propose_fix
  type: llm
  provider: openai
  model: gpt-4o
  prompt: "Write a fix for: {{stages.triage.output}}"

- id: approve_fix
  type: input
  message: |
    PROPOSED FIX:
    {{input}}

    Type "apply" to proceed, "skip" to close, or describe an alternative:
  next:
    - condition: "output.toLowerCase().startsWith('apply')"
      stage: apply_fix
    - condition: "output.toLowerCase().startsWith('skip')"
      stage: close
    - stage: revise_fix          # any other text → revise with feedback

Use aiac for planning and approval, then hand off implementation to Claude Code's interactive session. Claude Code has full filesystem and shell access — it can browse the codebase, run commands, and iterate autonomously.

→

→

→

  type: file
  action: write

aiac workflows run cleanly in CI — as long as you use no input stages or approve: true loops (those require a TTY). The pattern: the Action writes the PR diff to a temp file, runs aiac headlessly, and posts the compiled review as a PR comment.

Option A — Workflow Posts the Comment

The aiac workflow itself has a final post_comment shell stage. The Action just runs aiac and provides GITHUB_TOKEN + PR_NUMBER as env vars.

.github/workflows/pr-review.yml — Option A
name: PR Code Review

on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  review:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
      contents: read

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - uses: actions/setup-node@v4
        with:
          node-version: 20

      - name: Install aiac
        run: npm install -g @thegrotesk/aiac

      - name: Generate PR diff
        run: |
          git diff origin/${{ github.base_ref }}...HEAD \
            -- '*.ts' '*.tsx' '*.js' '*.jsx' '*.py' '*.go' \
            | head -c 60000 \
            > /tmp/pr_diff.txt

      - name: Run aiac PR review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
          GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
          PR_NUMBER: ${{ github.event.pull_request.number }}
        run: |
          aiac run workflows/ci/pr-review.yaml \
            --prompt "PR #$PR_NUMBER" \
            --fresh

Tips & Gotchas

CI-safe workflows

Workflows running in GitHub Actions must not contain input stages or loop stages with approve: true — both require a TTY. Create separate workflows/ci/ variants for automation and keep the interactive versions in workflows/dev/.

Diff size limits

Large PRs can exceed model context limits. The Action examples cap the diff at 60 KB with head -c 60000. For very large PRs, filter to specific file types or split into multiple aiac runs per directory.

bash — limit diff by path or size
# Only TypeScript files, max 60 KB
git diff origin/$BASE...HEAD -- '*.ts' '*.tsx' | head -c 60000 > /tmp/pr_diff.txt

# Exclude generated files and lock files
git diff origin/$BASE...HEAD \
  -- ':(exclude)*.lock' ':(exclude)dist/**' ':(exclude)*.min.js' \
  | head -c 60000 > /tmp/pr_diff.txt

Required GitHub secrets

SecretWhere to addNotes
ANTHROPIC_API_KEYRepo → Settings → Secrets → ActionsRequired for all LLM stages.
GITHUB_TOKENAuto-provided by GitHub ActionsNo setup needed. Grant pull-requests: write permission in the job.

Using --fresh in CI

Always pass --fresh in CI runs. Without it, aiac will look for a checkpoint from a previous run and prompt interactively — which hangs the job.

Avoid duplicate comments on re-runs

By default, each push creates a new comment. Use sticky-pull-request-comment (Option B) or add a shell stage that first deletes any existing bot comment before posting.

Use Case: Code Review Pipeline

Multi-model review — Sonnet analyzes, GPT-4o suggests fixes, Haiku rates severity, results saved to a timestamped report.

examples/code-review.yaml
name: Code Review Pipeline
stages:
  - id: analyze
    type: llm
    provider: anthropic
    model: claude-sonnet-4-6
    system: You are an expert code reviewer. Identify bugs, security issues, and quality problems.
    prompt: |
      Review this code:
      {{input}}
    max_tokens: 2048

  - id: suggest
    type: llm
    provider: openai
    model: gpt-4o
    prompt: "Suggest specific, actionable fixes for:\n{{input}}"
    max_tokens: 2048

  - id: save_review
    type: file
    action: write
    path: ./reviews/review-{{timestamp}}.md
    content: |
      ## Analysis
      {{stages.analyze.output}}

      ## Suggested Fixes
      {{stages.suggest.output}}

  - id: rate_severity
    type: llm
    provider: anthropic
    model: claude-haiku-4-5-20251001
    system: "Reply with exactly one word: critical, warning, or ok"
    prompt: "{{stages.analyze.output}}"
    max_tokens: 10
bash
cat src/auth.ts | aiac run examples/code-review.yaml --prompt "$(cat src/auth.ts)"

Use Case: Feature Development

Full AI-driven feature implementation: read the codebase, design a plan, loop through implement → apply → test, pause for human review after each iteration.

bash
cd /my/project
aiac run examples/develop-feature.yaml \
  --prompt "Add rate limiting to the /api/users endpoint"

The develop-feature.yaml workflow:

  1. Opus reads src/**/*.ts and writes an implementation plan
  2. A loop runs up to 5 iterations: GPT-4o implements → apply → typecheck → test
  3. After each iteration you see the output and can type feedback or done
  4. Haiku writes a 5-bullet summary when you exit the loop

Use Case: Incident Response

Paste an error or alert. The workflow triages severity, branches on critical vs. non-critical, proposes and applies a fix with human approval, writes a postmortem, and notifies Slack.

bash
aiac run examples/incident-response.yaml \
  --prompt "OOMKilled on api-server pod, 3 restarts in 10min, p95 latency spiked to 8s"

Key patterns used:

  • Conditional routing — Haiku classifies severity, critical path goes directly to escalation + Slack.
  • Human approval gate — engineer reviews triage + proposed fix before applying.
  • HTTP stage — Slack webhook notification using {{env.SLACK_WEBHOOK_URL}}.
  • File stage — incident log and postmortem saved with {{timestamp}} filenames.

Use Case: Content Pipeline

Research a topic, draft an article, critique it, rewrite based on the critique, then publish via API.

yaml — sketch
stages:
  - id: research
    type: llm
    provider: anthropic
    model: claude-opus-4-7
    prompt: "Research and outline an article about: {{input}}"

  - id: draft
    type: llm
    provider: openai
    model: gpt-4o
    prompt: "Write a 600-word article based on:\n{{input}}"

  - id: critique
    type: llm
    provider: anthropic
    model: claude-sonnet-4-6
    prompt: "Critique this draft for clarity, accuracy, and engagement:\n{{input}}"

  - id: rewrite
    type: llm
    provider: openai
    model: gpt-4o
    prompt: |
      Original draft:
      {{stages.draft.output}}

      Critique:
      {{stages.critique.output}}

      Rewrite incorporating the feedback.

  - id: publish
    type: http
    method: POST
    url: "{{env.CMS_API_URL}}/articles"
    headers:
      Authorization: "Bearer {{env.CMS_TOKEN}}"
      Content-Type: application/json
    body:
      title: "{{input}}"
      body: "{{stages.rewrite.output}}"
      status: draft
